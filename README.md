# BandSync

局域网多设备音频同步辅助工具，面向乐队排练、现场演出或需要多台 Android 设备同时播放不同音轨的场景。

核心思路：

- 服务器播放 `Server Output`，例如伴奏或主输出。
- 客户端播放 `Clients Output`，例如节拍器、提示音轨或伴奏加节拍器。
- `Clients Output` 会先整首下载到客户端本地缓存，再执行服务端命令。
- 播放过程中不做持续流式同步、不做持续 seek 纠偏，降低中间同步逻辑复杂度。

## 功能概览

- 服务器/客户端双模式。
- 服务器监听局域网端口，默认 `5777`。
- 客户端输入服务器 IP 和端口连接，IP 和端口会自动持久化。
- 支持选择本机播放音频 `Server Output`。
- 支持选择客户端播放音频 `Clients Output`。
- 支持从 `Server Output` 自动分析 BPM 并生成节拍 WAV 作为 `Clients Output`。
- 客户端整首缓存 `Clients Output` 后再播放。
- 服务端下发开始、暂停、停止命令。
- 支持两种开始方式：
  - 设备时间同步。
  - 网络信号同步。
- 显示客户端缓存状态。
- 显示客户端与服务器之间的信号延迟和设备时间差。

## 运行模式

### 服务器模式

服务器负责：

- 监听本机所有可用 IPv4 地址。
- 显示可供客户端连接的 IP 和端口。
- 选择并播放 `Server Output`。
- 选择、缓存并提供 `Clients Output` 给客户端下载。
- 统计可见客户端的连接状态、缓存状态、信号延迟和设备时间差。
- 下发开始、暂停、停止命令。

服务器播放规则：

- 点击开始前，会检查当前可见客户端是否已缓存当前版本的 `Clients Output`。
- 如果存在未缓存完成的客户端，不会下发开始命令。
- 暂停和停止命令会立即执行，不等待设备时间。

### 客户端模式

客户端负责：

- 输入服务器 IP 和端口连接服务器。
- 建立事件长连接，接收服务器推送的音频状态和播放命令。
- 下载并缓存整首 `Clients Output`。
- 缓存完成后执行服务器下发的开始、暂停、停止命令。
- 周期性测量并显示与服务器之间的信号延迟和设备时间差。
- 将缓存进度、延迟和时间差上报给服务器显示。

客户端不会：

- 边播边流式拉取音频。
- 持续按服务器进度 seek。
- 使用延迟测量做播放过程中的漂移纠偏。

## 同步方式

### 设备时间同步

设备时间同步用于尽量让服务器和客户端在同一个“服务器目标时间”开始播放。

流程：

1. 服务器下发一个未来约 3 秒的目标执行时间。
2. 客户端通过 `/time-sync` 多次测量 `服务端时间 - 本机时间`。
3. 客户端使用稳定后的时间差校正本机执行时间。
4. 到达校正后的本机执行时间后开始播放。

客户端执行时间计算方式：

```text
估算当前服务端时间 = 客户端本机时间 + 时间差
等待时间 = 服务端目标执行时间 - 估算当前服务端时间
```

如果命令到达时已经晚了，客户端会计算迟到时间，并从对应位置开始播放。

### 网络信号同步

网络信号同步用于更直接的控制体验。

流程：

1. 服务器通过事件长连接推送命令。
2. 客户端收到命令后立即执行。

这种方式更直观，但一致性取决于网络延迟和客户端接收命令的时机。

## 时间差和信号延迟

客户端使用类似 NTP 的四时间戳方式估算时间差：

```text
t1 = 客户端发送请求时间
t2 = 服务端收到请求时间
t3 = 服务端发送响应时间
t4 = 客户端收到响应时间
```

估算值：

```text
信号延迟 = (t4 - t1) - (t3 - t2)
时间差 = ((t2 - t1) + (t3 - t4)) / 2
```

其中：

- `时间差 > 0` 表示服务端时间比客户端快。
- `时间差 < 0` 表示服务端时间比客户端慢。

稳定策略：

- 初始连接后前 5 次快速采样，每 500ms 一次。
- 稳定后每 3 秒采样一次。
- 保留最近 9 个样本。
- 信号延迟使用中位数平滑。
- 时间差优先采用 RTT 最低的样本，再做平滑。
- 如果时间差突然跳变超过 1000ms，会直接采用新值，避免长期拖尾。
- 计算时间差时会扣除服务端处理耗时；在上下行延迟近似对称时，延迟会被四时间戳公式抵消。
- 真实网络上下行通常不完全对称，因此时间差仍会受延迟抖动影响。
- UI 会显示估算误差范围：以最低 RTT 样本的 `RTT / 2` 加上窗口内延迟抖动的一半作为保守提示。
- 使用设备时间同步开始播放时，客户端收到开始命令后会立即追加 2 次快速 time-sync，再按校正后的时间执行。

## 网络接口

服务器当前使用简单 HTTP 协议：

- `GET /events`  
  客户端事件长连接。服务端通过 NDJSON 持续推送音频状态、命令和保活事件。

- `GET /stream`  
  下载当前 `Clients Output`。客户端会整首缓存到本地。

- `GET /client-status`  
  客户端上报缓存状态、缓存进度、信号延迟和时间差。

- `GET /time-sync`  
  客户端测量时间差和信号延迟。

说明：

- 客户端不再轮询 `/state`。
- 播放命令通过 `/events` 长连接推送。
- 缓存进度由客户端在状态变化时主动上报。

## 构建

Debug APK：

```powershell
.\gradlew.bat assembleDebug
```

输出：

```text
app/build/outputs/apk/debug/app-debug.apk
```

Release APK：

```powershell
.\gradlew.bat assembleRelease
```

输出未签名 APK：

```text
app/build/outputs/apk/release/app-release-unsigned.apk
```

## 签名

当前项目未配置正式 release 签名。

本地测试时可以使用测试 keystore：

```text
app/bandsync-test-signing.keystore
```

已签名测试 APK 输出：

```text
app/build/outputs/apk/release/app-release-signed-testkey.apk
```

注意：

- 测试签名 APK 可用于安装测试。
- 正式发布或上架时，应使用你自己的正式 release keystore。
- 不要把正式 keystore 和密码提交到公开仓库。

## 使用建议

- 所有设备应连接到同一个局域网。
- 服务器端应优先使用稳定 Wi-Fi，避免热点频繁切换。
- 使用设备时间同步前，建议先等待客户端显示稳定的时间差。
- 如果设备系统时间差较大，建议开启 Android 的自动设置时间和自动设置时区。
- 如果更看重点击后的即时响应，可以选择网络信号同步。
- 如果更看重多设备开始时间一致性，可以选择设备时间同步。

## 当前限制

- 仅支持 Android 客户端。
- 没有使用后台服务，应用被系统杀死后连接会断开。
- 没有正式发布签名配置。
- 没有复杂的断点续传策略，通常依赖重新缓存。
- 设备时间同步依赖时间差估算质量，网络抖动很大时仍可能出现误差。
